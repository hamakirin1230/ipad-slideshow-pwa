# リスク登録簿

## 方針

このプロジェクトでは、「実装できるか」よりも「本番で事故らないか」を重視する。

特に、iPad本番再生、オフライン同期、Google Drive権限、GitHub Pages配信、誤操作防止を重点的に管理する。

## 初期リスク一覧

| リスク                                | 影響 | 初期対策                                                        | 状態  |
| ---------------------------------- | -- | ----------------------------------------------------------- | --- |
| iPadのIndexedDBデータが消える              | 高  | 本番前の再検証画面を用意し、同期済み状態を明示する                                   | 未対応 |
| GitHub PagesのbasePath設定でPWAが壊れる    | 中  | 初期からProject site前提で `basePath: "/ipad-slideshow-pwa"` を設定する | 対応中 |
| 動画容量が大きく同期に失敗する                    | 高  | 初期上限を明示し、同期失敗時の再試行と中断再開を設計する                                | 未対応 |
| Drive APIの権限設計が複雑化する               | 高  | `drive.file` 中心の最小権限に固定し、Drive全体アクセスを避ける                    | 未対応 |
| 本番中に誤操作する                          | 高  | 本番モード、操作ロック、削除操作の制限を用意する                                    | 未対応 |
| Service Workerのキャッシュで古い画面が残る       | 中  | 第1-1ではService Workerを入れず、後続でキャッシュ更新設計後に導入する                 | 対応中 |
| iPadのSafari通常タブで使われてしまう            | 中  | 正式利用はホーム画面追加PWA前提とし、READMEと画面上で案内する                         | 未対応 |
| Drive上のデータを誤って削除する                 | 高  | iPad側ではDrive削除不可。PC側の完全削除は2段階確認と名前入力を必須にする                  | 未対応 |
| 公開前のdraft manifestをiPadが読んでしまう     | 高  | iPadは `published.manifest.json` のみ読む設計にする                   | 未対応 |
| 同期中断後の不完全データを本番再生してしまう             | 高  | 同期完了・検証完了フラグを分け、未完了データは本番再生不可にする                            | 未対応 |
| Google Photosを正本DBのように扱ってしまう       | 中  | Googleフォトは取り込み元に限定し、Drive上の再生用素材とmanifestを正本にする             | 未対応 |
| 公開リポジトリに写真・動画・manifest・秘密情報を置いてしまう | 高  | GitHubにはソースコードと設計文書のみ置く。素材・manifest・秘密情報は置かない               | 未対応 |
| WindowsとMacBookで開発環境がずれる           | 中  | Windows第1ゴール完了後、第1.5としてMacBookでclone・起動・ビルドを再確認する           | 未対応 |
| iPadOSやSafariの仕様差分でPWA挙動が変わる       | 中  | iPadOS 17以上を正式対応とし、実機確認結果をdocsに追記する                         | 未対応 |
| 本番前に最新版へ同期できていない                   | 高  | 同期日時、manifestバージョン、オフライン再生テスト結果を再生画面に表示する                   | 未対応 |

## 第1-1時点での重点管理

第1-1では、以下を重点的に確認する。

* GitHub Pages Project site前提のパス設定
* Service Workerをまだ入れないこと
* READMEとdocsに未実装機能を明示すること
* 本番事故につながるリスクを早期に記録すること

## 後続で追加する項目
